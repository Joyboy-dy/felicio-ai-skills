## `app/`

Use `app/` for routing orchestration.

### Allowed files

- `page.tsx`
- `layout.tsx`
- `template.tsx`
- `loading.tsx`
- `error.tsx`
- `not-found.tsx`
- `route.ts`
- Route groups like `(marketing)`, `(auth)`, `(dashboard)`
- Route-local `_components/`
- Route-local `_data/`
- Route-local `_lib/`
- Route-local `_actions/` only when the action is truly specific to that route

### What to avoid

- Global components dumped into `app/`
- Large business logic inside `page.tsx`
- Direct complex database logic inside `page.tsx`
- Random `utils/`, `hooks/`, or `services/` inside routes without clear ownership

### The colocation rule

```txt
If it is used only by one route, colocate it inside that route.
If it is reused by multiple routes of one domain, move it to features/[domain].
If it is reused across unrelated domains, move it to components/base, lib, hooks, or server.
```

## `features/`

Use `features/` for business/domain modules.

### Example structure

```txt
features/
└── collections/
    ├── components/
    ├── actions/
    ├── hooks/
    ├── schemas/
    ├── types.ts
    ├── constants.ts
    ├── queries.ts
    └── index.ts
```

### When to create a feature

Use a feature when the code belongs to a domain such as auth, billing, teams, collections, courses, recipes, orders, workspace, notifications, or links.

### Feature rules

- A feature may import from `components/ui`, `components/base`, `lib`, `hooks`, `types`, `config`, and `server` when appropriate.
- A feature should not reach into another feature's private internals.
- If cross-feature access is needed, expose a minimal public API through `index.ts`.
- Do not create a feature for one tiny component unless the domain is expected to grow.

## `components/ui/`

Reserve this folder for UI primitives from shadcn/ui or equivalent generated library components.

### Allowed examples

```txt
components/ui/button.tsx
components/ui/dialog.tsx
components/ui/input.tsx
components/ui/dropdown-menu.tsx
components/ui/card.tsx
```

### Forbidden examples

```txt
components/ui/dashboard-card.tsx      ← domain-specific
components/ui/course-card.tsx         ← domain-specific
components/ui/billing-panel.tsx       ← domain-specific
components/ui/custom-navbar.tsx       ← layout, not a primitive
components/ui/base-button.tsx         ← custom wrapper, belongs in base/
```

The reason this separation matters is that `components/ui/` is often auto-generated by `npx shadcn@latest add`. Mixing custom code here creates merge conflicts and makes it unclear which files are safe to regenerate.

## `components/base/`

Use `components/base/` for user-created reusable primitives and design-system wrappers.

### Examples

```txt
components/base/page-header.tsx
components/base/empty-state.tsx
components/base/data-table.tsx
components/base/icon-button.tsx
components/base/section-shell.tsx
components/base/status-badge.tsx
```

### Good pattern

```tsx
import { Button as ShadcnButton } from "@/components/ui/button";

export function IconButton() {
  return <ShadcnButton />;
}
```

### Bad pattern

```tsx
// This belongs in features/billing/components/
export function SubscriptionUpgradeButton() {}
```

A base component should be generic. It should not know about one business domain.

## `server/`

Use `server/` for backend-only logic.

### Recommended structure

```txt
server/
├── db/
│   └── prisma.ts
├── repositories/
│   ├── user.repository.ts
│   └── link.repository.ts
├── services/
│   ├── user.service.ts
│   └── billing.service.ts
├── auth/
├── mail/
├── storage/
├── cache/
└── integrations/
```

### Rules

- **Repositories** talk to the database. They handle queries, inserts, updates, and deletes. They do not contain business rules.
- **Services** contain business rules and orchestration. They call repositories and coordinate between multiple data sources.
- Route handlers and Server Actions should call services instead of embedding business logic directly.
- Keep server-only code out of client components.
- If you catch yourself importing `prisma` directly in a `page.tsx` or an action with complex logic, extract it to a repository and call it through a service.

## `lib/`

Keep `lib/` small. It is for tiny, cross-cutting, stateless utilities.

### Allowed

```txt
lib/utils.ts
lib/cn.ts
lib/format-date.ts
lib/slugify.ts
```

### What does NOT belong in `lib/`

- Auth logic → `server/auth/` or `features/auth/`
- Billing logic → `server/services/` or `features/billing/`
- Database clients → `server/db/`
- Email clients → `server/mail/`
- Analytics → `server/integrations/` or `features/analytics/`
- Storage → `server/storage/`
- API clients → `server/integrations/`
- Feature-specific logic → `features/[domain]/`

If `lib/` has more than about 10–15 unrelated files, audit it. Something is being dumped there that has a better home.

## `hooks/`

Use root `hooks/` only for generic, domain-agnostic hooks.

### Allowed

```txt
hooks/use-media-query.ts
hooks/use-debounce.ts
hooks/use-mounted.ts
hooks/use-window-size.ts
hooks/use-clipboard-copy.ts
```

### Feature-specific hooks

Feature-specific hooks belong in their domain:

```txt
features/billing/hooks/use-subscription.ts
features/auth/hooks/use-session.ts
features/collections/hooks/use-collection-filter.ts
```

---

## `types/`

Use root `types/` only for truly global types — types that are shared across multiple unrelated domains.

### Allowed

```txt
types/global.d.ts
types/external-modules.d.ts
types/api-response.ts
```

### Feature-specific types

Feature-specific types belong in their domain:

```txt
features/billing/types.ts
features/auth/types.ts
```

---

## `config/`

Use for environment validation, app metadata, navigation config, feature flags, external service configuration, and constants that are not business entities.

### Examples

```txt
config/env.ts
config/site.ts
config/navigation.ts
config/feature-flags.ts
```
